# How to Start & End Allocate Delivery Resources

- ☐ A resource request is submitted jointly by **PM and Tech Lead** (Yes / No)
- ☐ Resource request is explicitly validated (Yes / No)
- ☐ Delivery Team document **or** Delivery Scenario exists (Yes / No)
- ☐ Delivery Team document **or** Delivery Scenario is accessible (Yes / No)
- ☐ Requested roles are explicitly defined (Yes / No)
- ☐ Quantity per role is explicitly defined (Yes / No)
- ☐ Required skills / stack per role are explicitly defined (Yes / No)
- ☐ Availability constraint is explicitly defined (Yes / No)
- ☐ No hiring or recruitment is expected as part of this workflow (Yes / No)

---

# **How to End Allocate Delivery Resources**

### Allocation

- ☐ All requested roles are fully allocated (Yes / No)
- ☐ No role is partially allocated (Yes / No)

### Team Definition

- ☐ Assigned project team is explicitly documented (Yes / No)
- ☐ Assigned team matches the validated resource request (Yes / No)

### Onboarding

- ☐ Developers are formally informed of their assignment (Yes / No)
- ☐ Developers received project brief (Yes / No)
- ☐ Developers are assigned initial tasks (Yes / No)

### Locking

- ☐ Developers are marked as allocated to the project (Yes / No)
- ☐ Developers are considered unavailable for reassignment without a new workflow (Yes / No)

Checklist

### **State 0 – Availability Assessment**

☐ Resource criteria are clearly defined (roles, skills, quantity, timing)

☐ Internal availability is identified against the criteria

☐ Hiring pipeline availability is checked (if applicable)

☐ Resources are clearly classified as **available** or **not available** (≤ 2 weeks)

---

### **State 1 – Team Selection**

☐ A concrete project team is proposed (names, roles, allocation, start dates)

☐ Technical fit of the proposed team is reviewed

☐ Team proposal is technically accepted

☐ A technically validated team is selected

---

### **State 2 – Assignment Confirmation**

☐ Developers are formally informed of their assignment

☐ Developers acknowledge the assignment

☐ Developers are onboarded to the project (context, access, tasks)

☐ Developer allocation is locked and not reassignable without a new workflow
