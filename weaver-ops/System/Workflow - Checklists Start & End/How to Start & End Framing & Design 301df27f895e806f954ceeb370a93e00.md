- ☐ Success criteria are explicitly written in the Expression of Need (Yes / No)

# **How to End Workflow: Framing & Design**

### Product & Scope

- ☐ Product documentation exists (Yes / No)
- ☐ Product documentation is validated (explicit validation recorded) (Yes / No)
- ☐ Functional scope is explicit and unambiguous in the product documentation (Yes / No)

### Design

- ☐ Design document exists (Yes / No)
- ☐ Animated prototype is complete and accessible (Yes / No)
- ☐ Design system is defined and documented (Yes / No)
- ☐ Design is validated (explicit validation recorded) (Yes / No)

### Technical

- ☐ Technical documentation exists (Yes / No)
- ☐ Technical documentation is consistent with validated product scope (Yes / No)
- ☐ Technical documentation is consistent with validated design (Yes / No)
- ☐ Technical documentation is validated (explicit validation recorded) (Yes / No)

### Delivery Preparation

- ☐ At least one delivery scenario is produced (Yes / No)
- ☐ Delivery scenarios are internally validated (Yes / No)
- ☐ Delivery scenarios are not shared with the client (Yes / No)

### Team Readiness

- ☐ Delivery team definition exists (Yes / No)
- ☐ Delivery team definition is validated (Yes / No)

States Checklist

### **State 0 – Need Clarification & Scope Intent Lock · Checklist**

☐ Client problem, objectives, and constraints explicitly documented

☐ Project direction agreed

☐ Success criteria defined

☐ Scope Intent explicitly validated by the client

---

### **State 1 – Product Definition**

☐ All features and user flows to be built are explicitly documented

☐ Expected behaviors, business rules, and edge cases are fully specified

☐ No open questions remain about product functionality

☐ Product scope is formally locked and change-controlled

☐ Product Documentation is completed and validated

☐ Product Roadmap is produced and consistent with the locked scope

---

### **State 2 – Design Definition**

☐ All user journeys and screens are fully designed with no missing UI states

☐ Animated prototype is completed and represents all main flows and interactions

☐ Design system is defined (core components, styles, interaction rules)

☐ Design strictly matches the validated product scope

☐ No missing, extra, or deviating functionality in the design

☐ Design is validated as a complete and coherent whole

---

### **State 3 – Technical Definition**

☐ Technical architecture and stack defined

☐ Data models and integrations specified

☐ Infrastructure and environments defined

☐ Technical Documentation validated

☐ Delivery Team defined

---

### **State 4 – Delivery Scenarios Definition**

☐ At least one executable internal delivery scenario is defined

☐ Delivery constraints (duration and capacity) are satisfied

☐ Delivery scenarios are validated and usable for sales decisions
