# Delivery Team

## **Purpose**

Define the minimal and evolvable delivery capacity required to operate continuous delivery safely, without committing to scope, velocity, or deadlines.

---

## **Content**

### **Minimal run capacity**

- Minimum delivery capacity required to start: **1 developer**
- This capacity is sufficient to begin continuous delivery
- Increasing capacity impacts cost, not commitments

---

### **Team roles**

For each role, specify:

- Role name
- Mandatory or optional
- Indicative seniority

Example:

- Backend Developer — mandatory — senior
- Frontend Developer — optional — mid
- QA — optional — mid
- DevOps — optional — on-demand

---

### **Technologies**

For each role, specify required technologies with a **mandatory or optional** flag.

- Technologies reflect the **existing stack**
- No learning, migration, or experimentation is implied

Example:

- Backend Developer
    - Node.js — mandatory
    - NestJS — mandatory
    - PostgreSQL — mandatory
- Frontend Developer
    - React — mandatory
    - TypeScript — mandatory

---

### **Frameworks & methodologies**

For each role, specify required frameworks, practices, or methodologies with a **mandatory or optional** flag.

- These define how the team operates, not what it delivers
- No certification requirement is implied

Example:

- Developers
    - Kanban — mandatory
    - Continuous delivery principles — mandatory
    - Code review workflow — mandatory
- QA
    - Exploratory testing — mandatory
    - Regression testing — mandatory

---

### **Constraints**

- No commitment on delivery speed
- No commitment on scope
- No commitment on deadlines
- Team composition does not imply delivery guarantees
- Any capacity change is handled outside the workflow

---

## **Outcome**

- Sales can price delivery capacity without interpretation
- Operations can trigger recruitment safely
- Continuous delivery can start with minimal capacity
